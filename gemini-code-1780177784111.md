# 🛠️ Backend Deployment & Troubleshooting Log (May 2026)

This document covers the infrastructure decisions, environment mismatches, and structural resolutions applied to **Vanguard Core Engine** during the stabilization of the backend service.

## 🏗️ Architecture Overview

The backend is built on a modern, high-performance stack configured for a local multi-tenant SaaS environment:
* **Runtime:** Node.js v24.14.0 (ES Modules natively enabled via `"type": "module"`).
* **Language:** TypeScript with strict type checking and `NodeNext` module resolution.
* **Server Framework:** Express.js (Asynchronous bootstrapping layer).
* **Database Layer:** PostgreSQL running in a Docker container (Port `5432`).
* **ORM:** Prisma v7.8.0 (Latest generation).
* **Data Layer Driver:** `@prisma/adapter-pg` + native `pg` connection pooling.
* **Local Cloud Infrastructure:** LocalStack simulating AWS S3 (Port `4566`).

## 🛑 Fixed Issues & Technical Breakdown

### 1. The Prisma 7 "Edge/Wasm" Constructor Trap
* **Symptom:** The server failed to start, throwing:
  `PrismaClientConstructorValidationError: Using engine type "client" requires either "adapter" or "accelerateUrl" to be provided to PrismaClient constructor.`
* **Root Cause:** In Prisma 7, the traditional native binary engines (`engineType = "binary"`) have been deprecated or stripped down from the core client initialization. By default, `new PrismaClient()` expects an Edge/Wasm infrastructure or an optimization proxy (Prisma Accelerate). Attempting to connect directly to a traditional local PostgreSQL instance without a driver adapter triggers this validation crash.
* **Resolution:** We shifted to the modern **Driver Adapter pattern**. We installed the native PostgreSQL client (`pg`) and the official Prisma adapter (`@prisma/adapter-pg`). We then initialized a classic connection pool and injected it explicitly into the Prisma constructor, bypassing the Edge runtime restrictions safely in local development.

### 2. Node Next Module Resolution (`ERR_MODULE_NOT_FOUND`)
* **Symptom:** Node threw errors stating it could not locate internal files (e.g., `src/app.ts` or `src/config/database`).
* **Root Cause:** Using `"moduleResolution": "NodeNext"` in `tsconfig.json` forces ECMAScript Modules (ESM) compliance. Under strict ESM rules, Node requires explicit file extensions on relative imports, and it expects a `.js` extension even if the source file is physically `.ts` (since it references the post-compilation target).
* **Resolution:** 1. Fixed relative imports inside `src/app.ts` to include extensions: `import prisma from './config/database.js';`.
  2. Standardized execution using `tsx watch` managed through NPM scripts to isolate Windows path mappings.

### 3. Ghost Directories & Caching in Windows File System
* **Symptom:** Code modifications inside `src/config/database.ts` were completely ignored, and errors still referenced old root paths.
* **Root Cause:** During the migration and sanitation of the `backend` folder, a duplicate `config/` directory was left directly under the root, causing the module loader to resolve properties out of sequence. Node 24's aggressive ESM thread caching was holding the broken client instance in memory.
* **Resolution:** Performed a hard cleanup of the project tree, ensuring that the **only** configuration layer lives inside `src/config/`. Purged stale runtime threads via PowerShell.

## 📋 Standardized Configuration Artifacts

### `backend/src/config/database.ts`
```typescript
import { PrismaClient } from '@prisma/client';
import { PrismaPg } from '@prisma/adapter-pg';
import pg from 'pg';
import * as dotenv from 'dotenv';

dotenv.config();

// Establish traditional connection pooling for PostgreSQL in Docker
const pool = new pg.Pool({ connectionString: process.env.DATABASE_URL });
const adapter = new PrismaPg(pool);

const prisma = new PrismaClient({
  adapter: adapter,
  log: process.env.NODE_ENV === 'development' ? ['query', 'error', 'warn'] : ['error'],
});

export default prisma;
